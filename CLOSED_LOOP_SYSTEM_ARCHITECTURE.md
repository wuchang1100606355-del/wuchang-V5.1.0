# 動態閉環消費價值分配系統架構（三階段版本）

## 系統階段

### 三階段設計

1. **開發階段（Development Phase）**
   - 可以通過談話調整系統
   - 可以通過談話修改規則
   - 可以通過談話測試功能
   - 靈活開發和調整

2. **交付階段（Delivery Phase）**
   - 所有規章正式交付系統
   - 通過電子簽章簽發執行
   - 建立規則層正式版本
   - 鎖定系統規則

3. **執行階段（Execution Phase）**
   - 規則只能通過電子簽章修改
   - 談話僅供查看和討論
   - 執行模組只讀取規則
   - 完全可究責

## 條約層（Treaty Layer）

### 條約設計原則

**重要原則：**
- 所有設計都會有條約
- 條約定義系統行為和責任
- 條約必須通過電子簽章生效
- 條約修改必須通過電子簽章

### 條約內容

**系統條約：**
- 規則層條約 - 定義規則修改流程和責任
- 執行層條約 - 定義執行模組行為和限制
- 介面層條約 - 定義介面使用規範和權限
- 分配條約 - 定義閉環消費分配規則
- 本益比條約 - 定義本益比計算和驗證規則
- 簽核條約 - 定義簽核權限和流程
- 科目條約 - 定義科目白名單和驗證規則
- 支出條約 - 定義支出上限和控制規則

### 條約生效流程

1. **條約起草**
   - 起草條約內容
   - 定義系統行為和責任

2. **條約審核**
   - 相關負責人審核
   - 確認條約內容

3. **電子簽章**
   - 具有可究責自然人的電子簽章
   - 記錄簽章時間、簽章人、簽章原因
   - 建立可追溯記錄

4. **條約生效**
   - 條約正式生效
   - 系統按照條約執行
   - 記錄條約歷史

## 創辦人專屬電子簽章與專利保護機制

### 創辦人專屬電子簽章

**重要原則：**
- 只有創辦人的電子簽章能修改系統
- 創辦人擁有系統修改的專屬權限
- 所有系統修改必須通過創辦人電子簽章

### 專利所有權保護

**創辦人權利：**
- 系統創辦人身份（永久）
- 專利所有權人（永久）
- 專利設計審查權（永久）
- 否決停用權（永久）

### 轉任或交由他人營運情況

**即使轉任或交由他人營運，創辦人依然擁有：**
- ✅ 系統創辦人身份（永久）
- ✅ 專利所有權人（永久）
- ✅ 專利設計審查權（永久）
- ✅ 否決停用權（永久）

### 專利設計審查權機制

**審查流程：**
1. 系統修改申請
2. 創辦人專利設計審查
3. 判斷是否違反專利設計
4. 審查結果：
   - 通過：允許修改
   - 否決：拒絕修改
   - 停用：停止系統運作

### 否決停用機制

**創辦人權限：**
- 可以否決任何違反專利設計的修改
- 可以停用違反專利設計的系統
- 否決和停用決定不可被他人覆蓋
- 所有否決和停用記錄永久保存

### 專利設計保護條約

**條約內容：**
1. 創辦人專屬電子簽章條約
   - 只有創辦人的電子簽章能修改系統
   - 創辦人擁有系統修改的專屬權限

2. 專利所有權保護條約
   - 創辦人擁有專利所有權（永久）
   - 即使轉任或交由他人營運，專利所有權不變

3. 專利設計審查權條約
   - 創辦人擁有專利設計審查權（永久）
   - 可以審查任何系統修改是否違反專利設計

4. 否決停用權條約
   - 創辦人擁有否決停用權（永久）
   - 可以否決任何違反專利設計的修改
   - 可以停用違反專利設計的系統
   - 否決和停用決定不可被他人覆蓋

## 專利保護對象：眾利價值

### 專利保護目的

**核心目的：**
- 專利用來保護眾利價值不被更動
- 確保核心價值分配原則永久不變
- 保護系統的核心理念和價值觀

### 眾利價值定義

**眾利價值核心原則（受專利保護）：**
1. **合理分配** - 所得分配合理
2. **合規分配** - 符合法規規範
3. **合正義分配** - 公平正義
4. **合永續分配** - 永續發展

### 專利保護範圍

**受專利保護的眾利價值要素：**
- 閉環消費分配原則（50% 消費循環，50% 組織運作）
- 本益比規定（必須過半，不含人事費）
- 無資本利得原則（所有盈餘歸零）
- 規模經濟效應保護（系統成本 10% 甚至更低）
- 科目白名單機制（非指定科目不得核銷）
- 支出上限機制（成本項目有上限）
- 簽核權限機制（總幹事 ,000，理事長 ,000）
- 動態重新分配機制（多餘資金重新分配）

### 專利設計審查標準

**審查標準：是否違反眾利價值**

創辦人審查任何系統修改時，判斷標準為：

1. **是否違反合理分配原則**
   - 修改是否破壞合理分配
   - 是否影響所得分配合理性

2. **是否違反合規分配原則**
   - 修改是否違反法規規範
   - 是否影響合規性

3. **是否違反合正義分配原則**
   - 修改是否破壞公平正義
   - 是否影響分配正義性

4. **是否違反合永續分配原則**
   - 修改是否影響永續發展
   - 是否破壞可持續性

5. **是否違反閉環消費分配原則**
   - 修改是否改變 50% 消費循環分配
   - 是否改變 50% 組織運作分配

6. **是否違反其他核心價值原則**
   - 本益比規定
   - 無資本利得原則
   - 規模經濟效應保護
   - 科目白名單機制
   - 支出上限機制
   - 簽核權限機制

### 專利保護條約（眾利價值保護版）

**條約核心內容：**

1. **眾利價值保護條約**
   - 專利用來保護眾利價值不被更動
   - 確保合理、合規、合正義、合永續分配原則永久不變
   - 保護系統的核心理念和價值觀

2. **創辦人專利審查權條約**
   - 創辦人擁有專利設計審查權（永久）
   - 審查標準：是否違反眾利價值
   - 可以否決任何違反眾利價值的修改
   - 可以停用違反眾利價值的系統

3. **眾利價值不可更動條約**
   - 眾利價值核心原則不可被更動
   - 任何試圖更動眾利價值的行為都將被否決
   - 創辦人擁有永久保護權

## 可究責機制（雙重可究責）

### 可究責分類

**可究責分成兩種：**
1. **營運可究責** - 針對系統運作、執行、日常操作的可究責
2. **設計可究責** - 針對系統設計、架構、規則制定的可究責

### 營運可究責（Operational Accountability）

**責任範圍：**
- 系統日常運作
- 執行層操作
- 日常分配執行
- 簽核作業
- 核銷作業
- 資料記錄
- 系統維護

**營運可究責人員：**
- 總幹事（日常運作負責人）
- 理事長（營運監督負責人）
- 會計人員（財務作業負責人）
- 系統管理員（系統維護負責人）

**營運電子簽章：**
- 總幹事電子簽章（日常運作）
- 理事長電子簽章（營運監督）
- 會計人員電子簽章（財務作業）
- 系統管理員電子簽章（系統維護）

### 設計可究責（Design Accountability）

**責任範圍：**
- 系統設計
- 架構制定
- 規則制定
- 條約制定
- 專利設計
- 眾利價值保護
- 系統修改

**設計可究責人員：**
- 創辦人（系統設計者、專利所有權人）

**設計電子簽章：**
- 創辦人專屬電子簽章（系統修改、規則制定、條約制定）
- 創辦人專利審查簽章（專利設計審查）
- 創辦人否決停用簽章（違反眾利價值時）

### 雙重可究責機制

**系統運作需要雙重可究責：**

1. **營運可究責**
   - 負責日常運作
   - 負責執行層操作
   - 負責簽核作業
   - 營運人員電子簽章

2. **設計可究責**
   - 負責系統設計
   - 負責規則制定
   - 負責專利保護
   - 創辦人專屬電子簽章

### 可究責記錄

**所有操作都必須記錄：**
- 營運可究責記錄（誰執行了什麼操作）
- 設計可究責記錄（誰設計了什麼規則）
- 電子簽章記錄（簽章時間、簽章人、簽章原因）
- 完全可追溯

## 可究責權責與法律責任

### 核心原則：有權就有責

**重要原則：**
- 可究責對象有權就有責
- 所有權限都伴隨相應責任
- 所有操作都有法律後果

### 設計可究責權責與法律責任

**設計可究責權限：**
- 可改動系統規則
- 可制定規則
- 可修改條約
- 可調整系統架構

**設計可究責責任：**
- 如規則造成損害，由設計人負責
- 法律究責亦同（設計人承擔法律責任）
- 必須確保規則符合法規
- 必須確保規則不違反眾利價值

**設計可究責法律責任：**
- 規則造成損害：設計人負責
- 法律究責：設計人承擔
- 損害賠償：設計人承擔
- 所有設計決策都有法律後果

### 營運可究責權責與法律責任

**營運可究責權限：**
- 可處理帳務
- 可做支出核定
- 可做財務修正
- 可執行日常運作

**營運可究責責任：**
- 若有財務問題，必須負擔損害賠償
- 法律責任亦同（營運人員承擔法律責任）
- 必須確保帳務正確
- 必須確保支出核定合規
- 必須確保財務修正正確

**營運可究責法律責任：**
- 財務問題：營運人員負擔損害賠償
- 法律責任：營運人員承擔
- 帳務錯誤：營運人員負責
- 支出核定錯誤：營運人員負責
- 財務修正錯誤：營運人員負責

### 可究責權責條約

**條約內容：**

1. **設計可究責權責條約**
   - 設計可究責可改動系統規則
   - 如規則造成損害，由設計人負責
   - 法律究責亦同（設計人承擔法律責任）
   - 所有設計決策都有法律後果

2. **營運可究責權責條約**
   - 營運可究責可處理帳務、做支出核定及財務修正
   - 若有財務問題，必須負擔損害賠償
   - 法律責任亦同（營運人員承擔法律責任）
   - 所有營運操作都有法律後果

3. **有權就有責條約**
   - 可究責對象有權就有責
   - 所有權限都伴隨相應責任
   - 所有操作都有法律後果
   - 完全可追溯和可究責

## 權利義務明細

### 設計可究責權利義務明細

#### 設計可究責權利（Rights）

1. **系統規則修改權**
   - 有權改動系統規則
   - 有權制定新規則
   - 有權修改現有規則
   - 有權刪除規則

2. **條約制定權**
   - 有權制定條約
   - 有權修改條約
   - 有權廢除條約

3. **系統架構調整權**
   - 有權調整系統架構
   - 有權修改系統設計

4. **專利設計審查權**
   - 有權審查專利設計
   - 有權否決違反專利設計的修改
   - 有權停用違反專利設計的系統

5. **眾利價值保護權**
   - 有權保護眾利價值不被更動
   - 有權否決違反眾利價值的修改

#### 設計可究責義務（Obligations）

1. **規則合規義務**
   - 必須確保規則符合法規
   - 必須確保規則不違反眾利價值
   - 必須確保規則符合專利設計

2. **損害賠償義務**
   - 如規則造成損害，必須負責
   - 必須承擔損害賠償責任

3. **法律責任義務**

   - 必須承擔法律責任

   - 有權執行核銷作業

5. **系統維護權**

   - 有權進行資料備份

   - 必須確保支出符合支出上限
   - 必須確保支出符合簽核權限

3. **財務修正正確義務**

   - 必須確保財務修正合規

   - 必須承擔法律究責

   - 必須記錄所有帳務處理
   - 必須記錄所有支出核定

   - 必須確保完全可追溯
